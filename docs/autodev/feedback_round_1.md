# 第 1 轮用户体验反馈

> **角色**：独立开发者 — 有自己的副业/开源项目的独立开发者，希望自动化产品迭代过程，减少重复工作
> **关注重点**：全面体验，覆盖所有功能，建立基线

## 总体印象

autodev 的核心理念非常打动我——"让 AI 替你做产品迭代"，这精准命中了独立开发者的痛点：我有一个 side project，但没有用户来给我提反馈，也没有精力去做系统性的用户体验审查。autodev 用 AI 模拟用户视角自动发现问题，这个切入点在同类工具中是独一无二的。

但在实际上手过程中，我遇到了不少摩擦。整个产品是"纯 Markdown Skill 文件"，零依赖、零代码，这既是优势也是劣势：安装确实简单（一行 curl），但缺乏可验证性——我无法判断安装是否真正成功，也无法在不启动 Claude Code/Copilot CLI 的情况下预览效果。文档虽然内容丰富，但信息组织上对新手不够友好，存在中英文内容不一致、版本选择指引不够清晰等问题。

## 功能逐项评测

### 功能 1：安装流程

**体验描述**：README 提供了一键 curl 安装命令，非常简洁。但作为第一次接触的开发者，我有几个困惑：

1. **v1 vs v2 选择困难**：README.md 第 33-44 行给出了对比表格，但对于一个首次用户来说，"你的工具是否支持 sub-agent" 这个判断条件本身就需要额外知识。我使用 Copilot CLI，文档说它支持，但没有告诉我怎么验证。
2. **安装验证不充分**：README.md 第 104 行提供了验证命令 `cat ~/.claude/skills/autodev/SKILL.md | head -3`，只检查了文件头是否为 `---`，这远不够——如果文件只下载了一半也会通过。
3. **源码安装指引有误**：README.md 第 100-101 行 `cp autodev/skill/SKILL.md ~/.claude/skills/autodev/SKILL.md` 复制的是 v1，但上面推荐的是 v2，容易误导。

**具体问题**：
- `README.md:100-101`：源码安装示例复制了 v1 的 `SKILL.md`，与推荐使用 v2 矛盾
- `README.md:104`：验证命令过于简陋，无法确认文件完整性

### 功能 2：初始化配置（config.yaml 生成）

**体验描述**：SKILL.md 第 26-67 行描述了首次使用的初始化流程。设计上需要通过 `ask_user` 工具逐个询问 5 个问题，这比较友好。但有几个问题：

1. **无默认值/示例提示**：询问"源代码目录"时没有给出示例（如 `src/`、`miniprogram/`），新用户可能不知道怎么回答。
2. **无配置校验机制**：用户回答完后直接生成 config.yaml，没有确认步骤。如果我不小心把 source_dirs 写错了，需要手动编辑文件。
3. **examples 目录的发现性差**：`examples/` 下有两个很好的配置示例（`config-miniprogram.yaml` 和 `config-react-webapp.yaml`），但在初始化流程中完全没有提及。

**具体问题**：
- `skill/SKILL.md:28-35`：初始化问题缺少输入提示和示例值
- `skill/SKILL.md:41-65`：生成 config 后没有让用户确认的步骤

### 功能 3：迭代执行流程（v1 单 Agent）

**体验描述**：SKILL.md 第 69-171 行定义了完整的三步迭代循环。逻辑清晰：Step 1 模拟用户 → Step 2 设计计划 → Step 3 实施代码。

1. **动态策略设计很棒**：第 92-102 行定义了 10 轮不同的关注策略（UX → 核心流程 → 边缘场景 → 性能 → 竞品对比 → 国际化 → 压力测试 → 安全 → 开发者体验 → 回归测试），覆盖面广且循序渐进。
2. **智能优先级调整**：第 143-145 行的"连续 2 轮未解决自动提升 P0"规则很实用。
3. **缺乏错误恢复机制**：如果 Step 3 实施代码时出了问题（比如引入编译错误），流程没有回滚机制。SKILL.md 第 155 行只说"跳过并记录原因"，但如果改动已经写入文件了呢？
4. **测试验证是条件性的**：第 157 行说"如果项目有测试，运行测试"，但没有说如果测试失败了该怎么处理——是回滚修改、跳过当前项、还是继续下一项？

**具体问题**：
- `skill/SKILL.md:154-158`：实施后的验证/回滚策略不完整，缺少测试失败时的处理指引
- `skill/SKILL.md:159`：git commit 是在"全部完成后"一次性提交，如果中途崩溃，之前的改动会丢失在 working tree 中

### 功能 4：迭代执行流程（v2 多 Agent）

**体验描述**：SKILL-multi-agent.md 是 v2 版本，核心创新是通过 sub-agent 实现 context 隔离。

1. **Orchestrator 模式清晰**：第 14-37 行清楚定义了编排者的职责和不做的事，角色划分合理。
2. **Agent 间通信仅靠文件系统**：第 26 行 "文件系统 = Agent 间通信通道"。这意味着如果 sub-agent 没有正确写入文件，后续 Agent 会拿不到输入。虽然第 171-173 行有校验步骤，但只检查了"文件已创建"和"非空"，没有验证内容格式是否正确。
3. **模型选择指南非常详细**：第 316-378 行的模型推荐表格和选型依据是一个亮点，但信息量太大，可能会让新手感到不知所措。建议先给一个"开箱即用"的默认方案，再展开详细解释。
4. **v1 回退机制不明确**：第 396 行提到"如果 task 工具不可用，回退到 v1 模式"，但没有说明如何检测、如何自动回退。

**具体问题**：
- `skill/SKILL-multi-agent.md:171-173`：Agent 输出校验不够严格，仅检查文件存在和非空
- `skill/SKILL-multi-agent.md:396`：v1 回退机制只有一句话描述，缺乏具体实现指引
- `skill/SKILL-multi-agent.md:267-269`：git commit 由 Orchestrator 执行，但代码改动在 sub-agent 中完成，如果 sub-agent 部分成功部分失败，commit 会包含不完整的改动

### 功能 5：多 Persona 支持

**体验描述**：支持多角色轮换（SKILL.md 第 187-204 行），使用取模算法 `personas[(N-1) % len(personas)]` 实现轮换。

1. **设计简洁有效**：对于大多数场景够用。
2. **缺少 persona 权重机制**：如果我想让"新手用户"出现频率高于"高级用户"，目前无法实现。
3. **示例 persona 偏向 C 端产品**：examples/ 中的两个示例都是面向终端用户的产品（记账本、任务管理），对于我这种做开发者工具/CLI 的场景，缺少合适的参考。

**具体问题**：
- `skill/SKILL.md:203`：轮换算法固定，无法自定义权重
- `examples/`：缺少面向开发者工具、API 服务、CLI 工具等 B 端/开发者场景的配置示例

### 功能 6：中途控制

**体验描述**：SKILL.md 第 207-212 行定义了 5 种中途控制命令（暂停、跳过、调整方向、加角色、看状态）。

1. **功能列表齐全**：覆盖了常见的中途干预需求。
2. **但纯靠 AI 理解自然语言**：没有明确的命令格式，全靠 AI 理解用户意图。如果我说"我觉得方向不太对"，AI 能正确识别为"调整方向"吗？
3. **v2 新增了"换模型"命令**（SKILL-multi-agent.md 第 389 行），但 v1 没有同步说明这个区别。

**具体问题**：
- `skill/SKILL.md:207-212`：中途控制的触发条件完全依赖自然语言理解，没有 fallback 或明确语法

### 功能 7：迭代完成后的文档处理

**体验描述**：SKILL.md 第 177-183 行提供了三种文档处理选项（保留并提交、保留但 gitignore、删除）。

1. **选项设计合理**：覆盖了主要场景。
2. **缺少默认推荐**：没有告诉用户哪个选项最常用或推荐。
3. **"删除"选项风险较高**：删除操作不可恢复（假设用户没有其他备份），应该有二次确认。

**具体问题**：
- `skill/SKILL.md:177-183`：三个选项没有标注推荐项
- `skill/SKILL.md:183`：删除操作缺少二次确认提示

### 功能 8：文档体系

**体验描述**：项目文档包括中文 README.md、英文 README.en.md、CONTRIBUTING.md。

1. **README 质量高**：结构清晰，有竞品对比、版本对比、配置示例、运行效果演示。
2. **中英文基本同步**：内容对齐做得不错。
3. **CONTRIBUTING.md 较简洁**：对于开源项目来说足够，但项目结构描述（第 42-51 行）没有提到 v2（SKILL-multi-agent.md）。
4. **缺少 CHANGELOG**：作为用户，我想知道版本历史和重要变更。
5. **缺少 Troubleshooting / FAQ 部分**：常见问题（如"为什么 AI 没有正确识别我的命令"、"config.yaml 格式错误怎么办"等）没有文档。

**具体问题**：
- `CONTRIBUTING.md:42-51`：项目结构中只列出了 `skill/SKILL.md`，遗漏了 `skill/SKILL-multi-agent.md`
- `README.md`：缺少 FAQ / Troubleshooting 章节
- 项目根目录：缺少 CHANGELOG.md

## 发现的 Bug（标注 P0/P1/P2，引用文件路径和行号）

1. **[P1] CONTRIBUTING.md 项目结构描述不完整** — `CONTRIBUTING.md:44`
   - 项目结构只列出了 `skill/SKILL.md`，缺少 `skill/SKILL-multi-agent.md`（v2 版本）。这会导致贡献者不知道 v2 的存在，提交 PR 时可能只更新 v1 而忘记同步 v2。

2. **[P1] README 源码安装命令与推荐版本不一致** — `README.md:100-101`，`README.en.md:99-100`
   - 源码安装示例 `cp autodev/skill/SKILL.md ~/.claude/skills/autodev/SKILL.md` 复制的是 v1，但文档上方明确推荐使用 v2（SKILL-multi-agent.md）。新用户按照指引操作会安装到非推荐版本。

3. **[P2] v2 SKILL 文件中 config.yaml 模板的 agents 配置注释中模型名称可能过时** — `skill/SKILL-multi-agent.md:88-91`
   - 注释中写死了具体模型名（如 `gemini-3-pro-preview`、`claude-sonnet-4.6`），这些模型名会随时间过时。建议注释说明"请参考下方模型选择指南获取最新推荐"。

4. **[P2] .autodev/config.yaml 被包含在仓库中但 .gitignore 未排除** — `.autodev/config.yaml`，`.gitignore:1-2`
   - `.gitignore` 只排除了 `.autodev/logs/`，但 `.autodev/config.yaml` 是存在于仓库中的（作为项目自身的配置）。这本身不是 bug，但对于将 autodev 应用到其他项目的用户，如果他们 clone 了 autodev 仓库来学习，可能会困惑为什么 `.autodev/config.yaml` 被提交了——因为文档说它是"自动生成"的。建议在 config.yaml 中加注释说明这是 autodev 项目自身的自迭代配置。

5. **[P1] v2 的 v1 回退机制缺乏实现细节** — `skill/SKILL-multi-agent.md:396`
   - 文档声称 "如果 task 工具不可用（宿主工具不支持 sub-agent），回退到 v1 模式（SKILL.md）在主 context 中内联执行"，但没有提供任何具体的检测逻辑或回退步骤。AI 在实际执行时很可能不知道如何回退，导致直接报错。

6. **[P2] README "30秒体验"和"快速开始"内容重复** — `README.md:47-56` vs `README.md:80-104`
   - "30 秒体验"章节和"快速开始 > 安装"章节包含几乎相同的安装命令，造成冗余。如果更新安装方式，需要同步修改两处，增加维护负担。

7. **[P2] CONTRIBUTING.md PR 规范只提到 SKILL.md** — `CONTRIBUTING.md:30`
   - "如果修改了 `skill/SKILL.md`，请同时更新 README 中的相关说明"，但没有提到修改 `skill/SKILL-multi-agent.md` 时也需要同步。两个 SKILL 文件有大量重叠逻辑（初始化流程、config 格式、persona 支持等），修改一个很可能需要同步另一个。

8. **[P1] v1 和 v2 的 Step 3 实施完成后缺少测试失败处理策略** — `skill/SKILL.md:157`，`skill/SKILL-multi-agent.md:254`
   - 两个版本都提到"如果项目有测试，运行测试确保不破坏现有功能"，但都没有定义测试失败时应该怎么办（回滚？继续？标记为失败？）。这是一个关键的流程缺失。

## 改进建议（按优先级排列）

### P0（关键体验问题）

1. **补充测试失败时的处理策略**
   - **文件**：`skill/SKILL.md` 第 154-158 行，`skill/SKILL-multi-agent.md` 第 248-256 行
   - **建议**：在 Step 3 实施要求中明确添加：如果测试失败，应 `git checkout` 回滚当前改进项的改动，记录为"实施失败"，继续下一项。
   - **理由**：没有回滚策略，一次失败的改动可能污染整个项目状态，后续轮次都基于错误的基础。

### P1（重要改进）

2. **修复源码安装指引与推荐版本不一致**
   - **文件**：`README.md` 第 99-101 行，`README.en.md` 第 98-100 行
   - **建议**：将源码安装命令改为复制 v2 文件 `SKILL-multi-agent.md`，并加注释说明如需 v1 则替换文件名。
   - **理由**：新用户按文档操作会安装到非推荐版本，与文档推荐矛盾。

3. **完善 v2 的 v1 回退机制**
   - **文件**：`skill/SKILL-multi-agent.md` 第 396 行
   - **建议**：添加具体的检测逻辑（如"尝试调用 task 工具，如果返回工具不可用错误，则切换为内联执行模式"），并给出回退后的行为说明。
   - **理由**：当前回退机制形同虚设，AI 缺乏可执行的指令。

4. **更新 CONTRIBUTING.md 中的项目结构和 PR 规范**
   - **文件**：`CONTRIBUTING.md` 第 30 行和第 42-51 行
   - **建议**：在项目结构中补充 `skill/SKILL-multi-agent.md`；在 PR 规范中加入"如修改 `skill/SKILL-multi-agent.md`，也需同步更新 README 和检查 SKILL.md 是否需要同步修改"。
   - **理由**：避免贡献者遗漏 v2 文件的同步更新。

5. **初始化流程增加输入示例和确认步骤**
   - **文件**：`skill/SKILL.md` 第 28-67 行，`skill/SKILL-multi-agent.md` 第 51-93 行
   - **建议**：每个 ask_user 问题附带示例值（如 `源代码目录（如 src/、miniprogram/）：`），生成 config 后增加一步确认（"以上配置是否正确？输入 '是' 继续，或说明要修改的部分"）。
   - **理由**：降低首次使用的认知负担，减少配置错误。

### P2（锦上添花）

6. **添加面向开发者工具的配置示例**
   - **文件**：`examples/` 目录
   - **建议**：新增 `config-cli-tool.yaml` 或 `config-developer-tool.yaml` 示例，展示 CLI 工具/API 服务/开发者库类项目的 persona 配置。
   - **理由**：当前示例都面向 C 端产品，独立开发者做的很多是开发者工具。

7. **精简 README 中的重复内容**
   - **文件**：`README.md` 第 47-56 行 vs 第 80-104 行
   - **建议**：将"30 秒体验"简化为仅 3 行（安装命令 + 使用命令 + 看 AI 工作），详细安装说明只保留在"快速开始"中，或让"30 秒体验"链接到"快速开始"。
   - **理由**：减少维护负担，避免两处不一致。

8. **添加 FAQ / Troubleshooting 章节**
   - **文件**：`README.md`
   - **建议**：在 README 末尾添加常见问题（如：安装后无反应怎么办、config.yaml 格式错误、AI 未正确识别触发词、迭代中途中断如何恢复等）。
   - **理由**：减少用户在 Issues 中重复提问。

9. **config.yaml 中 agents 注释添加"参考下方指南"提示**
   - **文件**：`skill/SKILL-multi-agent.md` 第 86-91 行
   - **建议**：将注释中写死的模型名改为引导用户查看模型选择指南部分，如 `# 详见下方「Agent 模型选择指南」了解推荐模型`（实际上第 87 行已有类似注释，但 88-91 行又写死了具体模型名，建议去掉或标注为"示例"）。
   - **理由**：模型名可能过时，硬编码在配置模板中会误导用户。

10. **为 .autodev/config.yaml 添加来源说明注释**
    - **文件**：`.autodev/config.yaml`
    - **建议**：在文件开头加注释说明这是 autodev 项目自身的自迭代配置（用于 dogfooding），并非通用模板。
    - **理由**：避免用户 clone 仓库学习时产生困惑。
