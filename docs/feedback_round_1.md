# autodev 第1轮用户试用反馈报告

**试用者身份**：一名独立开发者，有 3 年 shell 脚本经验，正在用 autodev 自动迭代自己的 React 项目。之前手动跟 Copilot 对话做迭代，现在想用 autodev 自动化这个过程。期望工具稳定可靠、配置简单、输出清晰

**试用时间**：2026-03-01  
**当前版本**：v0.1.0  
**测试环境**：macOS (bash 3.2.57)

---

## 总体印象

autodev 作为一个自动化迭代工具，概念新颖且实用。通过三步循环（反馈→计划→实施）将以往需要人工干预的产品迭代流程完全自动化，这对独立开发者来说很有价值。**评分：7.5/10**

**亮点：**
- 三步循环设计逻辑清晰合理
- 配置系统灵活（支持自定义 prompt 模板）
- 错误处理较为完善
- 自举能力强（能迭代自身）
- 单文件分发，无外部依赖

**主要问题：**
- 缺乏增量执行机制（无法从某步骤中断后继续）
- 配置验证不足
- 部分用户体验细节需优化

## 安装与初始化体验

### ✅ 优秀表现
1. **安装简单**：单文件 `autodev.sh`，只需添加到 PATH 即可
2. **初始化完整**：`autodev init` 一步生成所有必需文件（代码行 164-301）
3. **配置文件结构清晰**：注释详细，变量语义明确（代码行 176-211）
4. **prompt 模板设计合理**：三个阶段的模板分离良好，占位符系统完善（代码行 124-159）

### ⚠️ 需要改进  
1. **缺少配置验证**：init 后没有验证必填字段（如 PRODUCT_NAME="我的产品"）
2. **路径处理不够健壮**：`resolve_path()` 函数（代码行 55-57）对符号链接处理可能有问题
3. **重复初始化提示不够友好**：只是警告并退出，没有提供覆盖选项（代码行 168-171）

### 🧪 测试结果
- ✅ `autodev init /tmp/test_autodev` 成功创建所有配置文件
- ✅ 重复执行 init 正确拒绝并提示
- ❌ 对无权限目录的错误提示不够明确

## run 执行体验

### ✅ 优秀表现
1. **dry-run 模式很实用**：可以预览 prompt 而不实际执行（代码行 93-99）
2. **进度显示清晰**：banner 和步骤分隔符视觉效果良好（代码行 37-53）
3. **日志记录完善**：每个步骤都有独立日志文件（代码行 394, 404, 415）
4. **模型配置灵活**：支持通过 `-m` 参数切换模型（代码行 376）

### ⚠️ 需要改进
1. **缺乏步骤级重试**：如果步骤 2 失败，必须从步骤 1 重新开始
2. **copilot 失败处理简陋**：只是记录退出码，没有重试或降级机制（代码行 104-107）
3. **文件生成校验不足**：只检查文件是否存在，不验证内容质量（代码行 396-398）
4. **缺少进度状态持久化**：中途中断后难以恢复

### 🧪 测试结果
- ✅ `--dry-run` 正确显示渲染后的 prompt
- ✅ prompt 模板变量替换正确
- ⚠️ 无实际 copilot 调用测试（避免产生费用）

## config 配置体验

### ✅ 优秀表现
1. **变量语义清晰**：每个配置项都有明确用途和合理默认值
2. **示例配置丰富**：提供了小程序和 React 两种场景的配置示例
3. **prompt 模板系统强大**：支持 11 个占位符，覆盖大部分使用场景（代码行 127-141）
4. **渲染引擎高效**：使用 awk 一次性完成所有变量替换（代码行 127-159）

2. **SOURCE_DIRS 处理简单**：只是简单替换，没有路径标准化
3. **敏感信息保护不足**：config.sh 可能包含 API key 等，需要添加到 .gitignore

## 与手动迭代的对比

### 🎯 自动化价值
相比手动与 Copilot 对话迭代，autodev 提供了：

1. **一致性**：每次都按相同步骤执行，避免遗漏
2. **可重复性**：配置化的 prompt，结果更稳定  
3. **批量处理**：支持多轮迭代，节省人力
4. **历史追踪**：自动生成文档和提交记录
5. **标准化**：统一的反馈格式和改进流程

### 📊 效率对比

| 维度 | 手动迭代 | autodev | 改善程度 |
|------|---------|---------|----------|
| 时间投入 | 30-60分钟/轮 | 5-15分钟/轮 | 60-75% |
| 一致性 | 低（易遗漏） | 高 | 显著改善 |
| 可追溯性 | 差 | 优秀 | 显著改善 |
| 学习门槛 | 低 | 中等 | 略有增加 |

### 🔄 手动干预场景
autodev 无法完全替代手动迭代的场景：
- 需要复杂业务判断的决策
- 跨系统的架构调整
- 需要外部服务集成的改进
- 紧急 Bug 修复（需要实时反馈）

## Bug 列表

### P0 - 紧急修复
*（无发现）*

### P1 - 重要问题
1. **缺少 copilot 依赖检查**
   - **位置**：`run_copilot()` 函数（代码行 89-108）
   - **影响**：工具启动后才发现 copilot 不可用，用户体验差
   - **复现**：在没有 copilot 的环境运行 `autodev run`

2. **缺少增量执行能力**
   - **位置**：`cmd_run()` 函数（代码行 356-430）
   - **影响**：步骤失败后无法从中途恢复，必须重头开始
   - **复现**：step 2 失败，无法跳过 step 1 重新执行

### P2 - 体验问题
1. **配置验证不足**
   - **位置**：`cmd_init()` 函数没有后续校验
   - **影响**：用户可能忘记编辑配置文件
   - **建议**：run 前检查是否还是默认占位符值

2. **错误信息可读性**
   - **位置**：`resolve_path()` 函数（代码行 56）
   - **影响**：错误信息过于技术化
   - **建议**：提供更友好的错误描述

## 改进建议（按优先级排序）

### 🔴 P0 - 立即修复
*（无发现严重阻塞问题）*

### 🟡 P1 - 重要改进
1. **增加前置条件检查**
   ```bash
   # 在 cmd_run() 开头添加
   check_prerequisites() {
     command -v copilot >/dev/null || die "Copilot CLI not found"
     git rev-parse --git-dir >/dev/null 2>&1 || die "Not a git repository"
   }
   ```

2. **实现增量执行机制**
   ```bash
   # 支持 --resume-from 参数
   autodev run . --resume-from step2
   ```
   - 需要在 logs 中记录步骤完成状态
   - 检查上一轮的输出文件是否存在

3. **改进错误处理**
   - copilot 失败时提供重试选项
   - 网络错误时自动重试机制
   - 更友好的错误消息

4. **配置验证器**
   ```bash
   validate_config() {
     [[ "$PRODUCT_NAME" != "我的产品" ]] || warn "请编辑 PRODUCT_NAME"
     [[ -d "$SOURCE_DIRS" ]] || warn "SOURCE_DIRS 路径不存在"
   }
   ```

### 🔵 P2 - 体验优化
1. **改进 status 命令**
   - 显示每轮的执行状态（成功/失败/进行中）
   - 显示最后执行时间
   - 提供失败轮次的错误信息摘要

2. **增强日志系统**
   - 支持不同日志级别（DEBUG/INFO/WARN/ERROR）
   - 提供 `--verbose` 参数控制输出详细程度
   - 日志轮转避免磁盘占满

3. **配置模板系统**
   ```bash
   autodev init . --template react  # 使用预设模板
   autodev init . --template miniprogram
   ```

4. **自举能力增强**
   - 检测到自己在自举时提供特殊提示
   - 避免无限递归的安全机制

5. **跨平台兼容性**
   - 将 `seq` 替换为 bash 原生循环（兼容性更好）
   - 添加 Linux/macOS 的功能差异说明

### 🟢 P3 - 未来特性
1. **Web 界面**：提供可视化的迭代历史查看
2. **插件系统**：支持自定义步骤和 hook
3. **多模型并行**：同时使用不同模型进行反馈收集
4. **集成测试**：自动运行测试套件验证改进效果

## 结论

autodev 是一个具有创新性和实用价值的工具，基本实现了自动化迭代的核心功能。虽然存在一些体验细节问题，但整体架构合理，扩展性良好。

**推荐指数**：⭐⭐⭐⭐☆（4/5）

**使用建议**：
1. 适合有一定 shell 脚本经验的开发者
2. 建议先在小项目上试用，熟悉流程后再用于重要项目
3. 结合手动迭代使用，在复杂决策场景下仍需人工干预

**预期改进效果**：
实施 P1 优先级改进后，工具可用性将显著提升，可以推荐给更多开发者使用。
